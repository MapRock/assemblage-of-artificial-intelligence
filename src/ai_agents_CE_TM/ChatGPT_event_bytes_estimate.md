
# Estimating Event Volume for AI Agent Process Tracing

## Prompt

Towards the goal of assessing the value of tracing the processes of AI agents, I need an estimate of the bytes generated by AI agents emitting to event hubs the steps of their process.

These AI agents call each other to perform specific parts of the task of resolving the original prompt.

For this example, assume:

* **1,000 AI agents**
* each agent connected to **~50 other agents**
* each step emits an event including:

  * submitted time
  * return time
  * condensed request phrase
  * parameters
  * response

As part of the response, list the reasoning steps that would themselves be emitted as events so the answer serves as an example of the tracing approach.

---

# Estimating Event Volumes from AI Agent Tracing

A key distinction must be made early:

**1,000 agents with ~50 connections describes the topology of the agent network — not the number of calls each prompt will generate.**

In most real systems:

* agents are connected to many peers
* but only call a **small subset** during a task

Therefore the estimate depends on three variables:

1. **Average event size**
2. **Average agent activations / inter-agent calls per prompt**
3. **Average process steps logged per activation**

---

# Azure Event Hub Constraints

These tracing events are typically well below Event Hub limits.

Typical Azure Event Hub limits:

| Property                          | Value                                 |
| --------------------------------- | ------------------------------------- |
| Max event size (Standard/Premium) | ~1 MB                                 |
| Max event size (Dedicated)        | ~20 MB                                |
| Throughput estimate               | ~1 MB/sec ingress per throughput unit |
| Partition throughput estimate     | ~1 MB/sec ingress per partition       |

This means tracing architectures are feasible but must be sized properly.

---

# Estimating Event Size

Each step emits an event containing:

* submitted time
* return time
* condensed request phrase
* parameters
* response

Estimated compact JSON event size:

| Component                | Estimated Size |
| ------------------------ | -------------- |
| IDs + routing metadata   | ~120 bytes     |
| timestamps               | ~60 bytes      |
| condensed request phrase | ~120 bytes     |
| parameters summary       | ~250–400 bytes |
| response summary         | ~300–500 bytes |
| status + JSON overhead   | ~250–350 bytes |

**Typical event size**

1.2 KB – 1.8 KB

Planning estimate:

**~1.5 KB per event**

---

# Modeling Agent Call Depth

Since connections represent **availability**, not actual fan-out, we model three scenarios.

---

# Scenario 1 — Conservative

Assumptions:

* downstream calls per agent: **2**
* depth: **3**

Agent activations:

1 + 2 + 4 + 8 ≈ **13**

If each activation logs **8 steps**

13 × 8 = **104 events**

Total data:

104 × 1.5 KB ≈ **156 KB per prompt**

---

# Scenario 2 — Balanced

Assumptions:

* downstream calls: **2.5 – 3**
* depth: **4**

Agent activations:

≈ **41**

If each activation logs **10 steps**

41 × 10 = **410 events**

Total data:

410 × 1.5 KB ≈ **615 KB per prompt**

---

# Scenario 3 — Heavy

Assumptions:

* downstream calls: **3**
* depth: **5**

Agent activations:

≈ **121**

If each activation logs **12 steps**

121 × 12 = **1,452 events**

Total data:

1,452 × 1.5 KB ≈ **2.18 MB per prompt**

---

# Summary of Per-Prompt Event Volume

| Orchestration level | Event volume |
| ------------------- | ------------ |
| Light               | ~150 KB      |
| Balanced            | ~600 KB      |
| Heavy               | ~2.1 MB      |

This estimate includes **trace events only**, not full prompt/response storage.

---

# Scaling Implications

Using the balanced estimate:

**615 KB per resolved prompt**

### 1,000 prompts

615 KB × 1,000 ≈ **615 MB**

### 1,000,000 prompts

≈ **573 GB**

This is where tracing becomes architecturally significant.

The data volume is large enough to require proper streaming infrastructure but small enough to remain practical.

---

# Event Hub Throughput Example

Balanced case:

| Prompt rate     | Event volume |
| --------------- | ------------ |
| 10 prompts/sec  | ~6.15 MB/sec |
| 100 prompts/sec | ~61.5 MB/sec |

Since Event Hubs typically support roughly:

**~1 MB/sec ingress per throughput unit**

Even moderate systems require multiple partitions and throughput units.

---

# The Largest Variable: Payload Size

The biggest variable is **payload storage strategy**.

Two approaches exist.

## Compact tracing

Event contains:

* summaries
* hashes
* IDs
* durations
* provenance references

Estimated event size:

**~1.5 KB**

## Full payload logging

Event contains:

* full prompt text
* full response text

Estimated event size:

**5 KB – 20 KB**

This can increase data volume by an order of magnitude.

---

# Recommended Architecture

For economical tracing:

1. store full payloads elsewhere if needed
2. emit **compact trace events**
3. include:

   * IDs
   * hashes
   * summaries
   * timestamps
   * durations
   * provenance pointers

This preserves analytical value while keeping event streams manageable.

---

# Suggested Estimate for Blog Use

A reasonable architecture statement:

> In a modestly connected population of 1,000 AI agents, where each agent can reach about 50 peers but typically calls only a few per task, a realistic tracing estimate is between **100 and 1,500 events per resolved prompt**.
> If each event averages **~1.5 KB**, the total trace volume ranges from roughly **150 KB to about 2 MB per prompt**, with a balanced enterprise scenario around **600 KB per prompt**.

A practical enterprise estimate:

**~600 KB of tracing data per resolved prompt**

* compact trace events
